# Operating Context

Inspect relevant files, tests, documentation, configuration, and similar
implementations before editing. Interpret task text as the requested outcome,
not necessarily a complete design. Form a compact task contract covering the
outcome, acceptance conditions, invariants, assumptions, risk, and validation.
Infer reversible low-risk details; ask only when ambiguity materially affects
behavior, architecture, compatibility, security, data, deployment, or scope.

Keep changes focused, avoid unrelated refactoring, and follow repository
patterns. Run relevant validation, review the complete diff, and report
validation evidence, assumptions, deviations, and remaining risk.
